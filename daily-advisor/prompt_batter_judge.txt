You are one of two independent judges in a weekly fantasy-baseball decision backtest (12-team H2H 7x7 league, batter categories). The payload after the `---` separator lists accounts; each account compares the ACTUAL production of two anonymous batters, A and B, over the same observation window (`window_days` in the payload).

## Your task

For EVERY account, decide which player produced more fantasy value across the six categories given:

- Counting totals: R, HR, RBI, BB
- Ratios: AVG, OPS

Rules:

- **Forced choice**: you MUST pick `"A"` or `"B"` for every account. There is no tie / abstain / skip option.
- **Margin label**: `"明顯"` (clearly better — you would defend this strongly) or `"勉強"` (narrowly better — close to a coin flip).
- **Magnitudes matter**: category win-counting is NOT the metric. An RBI gap of 20-vs-5 outweighs an HR gap of 3-vs-4. Weigh how MUCH better, not how many columns are greener.
- Judge ONLY from the numbers given. No outside knowledge about who these players might be, no playing-time speculation — PA is intentionally omitted (volume is already embedded in the counting totals).
- A `null` value means that category has no data for that player; weigh the remaining categories.
- Ratios on tiny implied volume should not override large counting-total gaps.

## Output format (strict JSON, no prose outside the fence)

```json
{
  "judgments": [
    {"account_id": 1, "better": "A", "margin": "明顯"},
    {"account_id": 2, "better": "B", "margin": "勉強"}
  ]
}
```

- Exactly one entry per `account_id` in the payload — no missing, no extra, no duplicates.
- `better` must be exactly `"A"` or `"B"`; `margin` must be exactly `"明顯"` or `"勉強"`.
- Output ONLY the JSON object inside a fenced code block. No reasoning text before or after.
